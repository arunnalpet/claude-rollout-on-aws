Subscribing to Claude for Enterprise on AWS Marketplace

Choose your subscription path to get started with Claude Enterprise on AWS. All paths lead to the same product — the difference is pricing and procurement process.

1 Self-Serve via AWS Marketplace

  • Minimum 20 seats required
  • Single seat type: Enterprise seat (includes Claude web, desktop, mobile + Claude Code)
  • Billed annually, per user per month
  • Add seats anytime — prorated for remainder of billing cycle, charged immediately
  • Seat reductions only take effect at annual renewal

2 Adding Incremental Seats

  • Go to Organization settings → Billing
  • Click the pencil icon under Seats
  • Enter the number of additional seats needed
  • Review the prorated charge and confirm
  • Alternatively: a seat is auto-purchased when inviting a new member if none are available

3 Private Offer

  • Work with your Anthropic account team to request a private offer
  • Private offer appears in AWS Marketplace → Manage subscriptions
  • Review terms, pricing, and seat count in the offer details
  • Accept the offer in AWS Marketplace to activate your subscription
💡 Note: Once subscribed through any approach, the Primary Owner receives a setup email from Anthropic to create the workspace. Proceed to the "Setup Claude for Enterprise (with Entra ID)" tab for next steps.

Pre-requisites

This guide assumes Microsoft Entra ID as your identity provider (IdP). Ensure the following teams and decisions are in place before proceeding.

  • Infra / Cloud team — will own the AWS Marketplace subscription and serve as the Primary Owner of the Claude workspace
  • Identity team — manages Microsoft Entra ID; will configure SSO (SAML 2.0) and optionally SCIM provisioning
⚠️ Identify your Primary Owner: There can be only one Primary Owner per workspace. This person is the super-user with full billing and admin control. You can have multiple Admins, but only one Primary Owner. Choose carefully — this should be someone from your Infra/Cloud team.

🔐 Identity Team Preparedness

Brief your Identity team on what they'll need to do. They can start prepping in parallel while the Infra team handles subscription + Primary Owner setup:

  • Domain verification — add a DNS TXT record for your company domain (required before SSO can be enabled)
  • Enterprise Application — create a new Enterprise App in Entra ID for Claude (SAML 2.0)
  • SCIM provisioning (optional) — requires Entra ID P1 or P2 license; enables automatic user sync
  • User/Group assignment — identify which Entra groups or users will be assigned access to Claude

Step 1: Primary Owner Setup

The admin contact from the deal receives a setup email from Anthropic to create the workspace.

  • Go to https://claude.ai/login
  • Create your workspace name — this is your org's Claude environment
  • The Primary Owner account uses magic-link (passwordless email) login until SSO is configured

Step 2: Configure SSO (Single Sign-On)

Set up SAML 2.0 with Microsoft Entra ID. Expand each sub-step below — note the owner badge indicating who performs each action.

▶ 2a. Verify your domain in Claude Primary Owner
  • Go to Claude Admin → Identity and access
  • Add your company domain and follow the verification steps
  • Requires adding a DNS TXT record (coordinate with your DNS admin)
▶ 2b. Start SSO setup flow in Claude Primary Owner
  • In Identity and access settings, start the SSO configuration
  • Note the Entity ID and Reply URL (ACS URL) — share these with your Identity team
  • Keep this flow open while the Identity team configures Entra
▶ 2c. Create Enterprise Application in Entra Identity Team
  • In Entra Admin Center → Enterprise applications → New application
  • Search for "Claude" in gallery, or create your own app named "Claude"
  • Select "Integrate any other application you don’t find in the gallery"
▶ 2d. Configure SAML settings Identity Team
  • Go to Single sign-on → SAML
  • In Basic SAML Configuration, enter the Entity ID and Reply URL from step 2b
  • Set Sign-on URL: https://claude.ai/login
  • In Attributes & Claims, set email claim to user.mail
  • Download the Federation Metadata XML
▶ 2e. Upload metadata & complete in Claude Both Teams
  • Identity team shares the Federation Metadata XML with Primary Owner
  • Primary Owner uploads it in the Claude SSO setup flow
  • Complete the connection wizard in Claude
▶ 2f. Configure SCIM provisioning (optional) Identity Team
  • Requires Entra ID P1 or P2 license
  • In the app → Provisioning → Set mode to Automatic
  • Enter Tenant URL and Secret Token from Claude setup flow
  • Ensure SCIM email attribute matches SSO email claim (user.mail)
  • Set Provisioning Status to On
▶ 2g. Assign users/groups & verify Identity Team
  • In Users and groups, assign people or groups who should access Claude
  • Trigger a provisioning cycle (if SCIM enabled) and confirm users appear in Claude
  • Have a test user complete SSO login → verify they land in the correct workspace
📖 Detailed guide: Microsoft Entra ID SSO Setup

Step 3: Invite Users

How users get access depends on whether SSO is configured yet.

  • Pre-SSO: Manually invite early users via the Admin Console (they'll use magic-link login)
  • Post-SSO: End users go to claude.ai/login → enter company email → get redirected through your corporate IdP automatically
  • With SCIM: Users are auto-provisioned when assigned in your IdP — no manual invites needed
  • With JIT (Just-in-Time): Users are created on first SSO login if they match the verified domain
✅ Done! Your Claude Enterprise workspace is now configured. Users on your domain can access Claude at claude.ai/login using company credentials.

Claude Code & Claude Cowork via Amazon Bedrock

Deploy Claude Code (CLI/IDE AI coding assistant) and Claude Cowork (desktop AI companion) through Amazon Bedrock. All data stays within your AWS boundary — Anthropic never sees prompts or completions.

💻 Claude Code

  • AI coding assistant (CLI/IDE-integrated) — code generation, review, debugging, refactoring
  • Uses AWS SDK credential chain to authenticate to Bedrock
  • For development teams

🖥️ Claude Cowork

  • AI desktop companion app — delegates long-running tasks, works with local files, MCP connectors
  • In-app AWS Sign-in via IAM Identity Center (no CLI needed on end-user machines)
  • For all teams — document analysis, research, workflow automation

Architecture Principles

  • Per-team dedicated AWS accounts for cost isolation and independent governance
  • Centralized identity via IAM Identity Center federated with Microsoft Entra ID
  • Data stays within AWS boundary — Anthropic never sees prompts or completions
  • Team-specific model access and configuration profiles
  • CloudTrail audit logging with per-user attribution

AWS Organizations Setup

  • AWS Organizations must be configured with a well-defined OU structure
  • Option A: Create a dedicated OU for Claude workload accounts (e.g., OU: AI-Assistants) — simplifies SCP inheritance
  • Option B: Place Claude accounts within existing team/BU OUs — aligns with current org structure but requires per-account SCP attachment
  • Each team gets a dedicated AWS account
  • Management account retains central governance and SCP attachment

IAM Identity Center Configuration

  • Enable IAM Identity Center in the management account (or delegated admin)
  • Configure Microsoft Entra ID as external identity source — SAML 2.0 federation + SCIM provisioning
  • Create Entra ID Security Groups per team (e.g., SG-Claude-TeamAlpha) for SCIM sync
  • Set session duration to 8-12 hours for permission sets (controls re-auth frequency)

Service Control Policies (SCPs)

Apply these SCPs to the Claude OU to enforce guardrails across all team accounts:

▶ Restrict Bedrock to Approved Models Only
  • Deny bedrock:InvokeModel unless InferenceProfileArn matches us.anthropic.claude-*
  • Prevents teams from invoking non-Claude or unapproved models
JSON
{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "DenyNonClaudeModels",
    "Effect": "Deny",
    "Action": ["bedrock:InvokeModel", "bedrock:InvokeModelWithResponseStream"],
    "Resource": "*",
    "Condition": {
      "StringNotLike": {
        "bedrock:InferenceProfileArn": [
          "arn:aws:bedrock:*:*:inference-profile/us.anthropic.claude-*",
          "arn:aws:bedrock:*:*:application-inference-profile/*"
        ]
      }
    }
  }]
}
▶ Deny Model Access Changes
  • Deny bedrock:PutUseCaseForModelAccess except for CentralAdminRole
  • Prevents teams from enabling additional models without central approval
JSON
{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "DenyModelAccessChanges",
    "Effect": "Deny",
    "Action": ["bedrock:PutUseCaseForModelAccess"],
    "Resource": "*",
    "Condition": {
      "StringNotLike": {
        "aws:PrincipalArn": "arn:aws:iam::*:role/CentralAdminRole"
      }
    }
  }]
}
▶ Enforce Region Restriction
  • Deny bedrock:* unless aws:RequestedRegion is in approved list (e.g., us-east-1, us-west-2)
  • Adjust regions based on data residency and latency requirements
JSON
{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "DenyNonApprovedRegions",
    "Effect": "Deny",
    "Action": ["bedrock:*"],
    "Resource": "*",
    "Condition": {
      "StringNotEquals": {
        "aws:RequestedRegion": ["us-east-1", "us-west-2"]
      }
    }
  }]
}

Bedrock Service Limits & Quotas

For each new team account, configure:

  • Request appropriate model invocation quotas (TPM) based on team size
  • Set up AWS Budgets with alerts at 80% and 100% thresholds
  • Configure CloudWatch alarms for throttling (InvocationThrottles metric)
  • Enable Bedrock Invocation Logging to S3 for audit trail
Team Size Recommended TPM Monthly Budget Alert
Small (5-15 users)100K-500K TPM$2,000-$5,000
Medium (15-50 users)500K-2M TPM$5,000-$20,000
Large (50+ users)2M+ TPM$20,000+

Shared Account vs. Dedicated Account

If a team plans to use an existing AWS account with Bedrock workloads:

⚠️ Key Risks of Sharing: Quota contention, cost attribution complexity, budget alert noise, broader security blast radius.
  • Review current quota utilization (CloudWatch: InvocationCount, TokenCount, InvocationThrottles)
  • Assess whether remaining headroom can absorb expected Claude traffic
  • If quota headroom insufficient or workloads are mission-critical → create a separate dedicated account
  • If sharing is acceptable → request quota increases + set up Application Inference Profiles for cost separation
💡 Best practice: Provision a dedicated AWS account per team. Eliminates quota contention, simplifies cost tracking, and ensures precise SCP scoping.

Anthropic Use Case Submission

One-time per account — required before Claude models can be invoked via Bedrock.

📋 Why is this needed? Anthropic requires every AWS account to submit a brief use-case form before enabling access to Claude models on Bedrock. This is a compliance and responsible-use requirement from Anthropic — it helps them understand how their models are being used and ensures alignment with their Acceptable Use Policy. The form is lightweight (company name, website, intended use case) and access is typically granted immediately upon submission.
▶ Option A: Via AWS Console
  • Navigate to Bedrock console > Model catalog
  • Select any Anthropic model and complete the use case form
  • Access granted immediately after submission
▶ Option B: Via CLI (recommended for automation)
  • Create form.json with company details and use case description
  • Run the CLI command below (must target us-east-1)
  • Submitting principal needs bedrock:PutUseCaseForModelAccess IAM permission
BASH
# Create the form data JSON
cat < form.json
{
  "companyName": "YOUR_COMPANY",
  "companyWebsite": "https://www.example.com",
  "intendedUsers": "0",
  "industryOption": "Customer Service",
  "otherIndustryOption": "",
  "useCases": "AI-powered productivity tools."
}
EOF

# Submit (must be us-east-1, must Base64 encode)
aws bedrock put-use-case-for-model-access \
  --form-data $(cat form.json | base64) \
  --region us-east-1

  • Prerequisites: AWS CLI v2, Claude Code CLI (npm install -g @anthropic-ai/claude-code), network access to Bedrock + Identity Center

▶ 3.5 VS Code Extension Developer

Standard Operating Procedures

Repeatable processes for common administrative actions.

▶ SOP-1: Onboard New User to Existing Team Identity Team
  • Add user to Entra ID security group → SCIM auto-syncs within 40 min
  • Deploy Claude tools to user machine (CLI + config or Cowork + managed config)
  • User acceptance test → verify end-to-end flow
  • Send onboarding email with usage guidelines
▶ SOP-2: Onboard New Team Cloud Platform + Identity
  • Create dedicated AWS account under Claude OU
  • Submit Anthropic use case form
  • Create Entra ID group + configure SCIM
  • Create permission set + assign group to account
  • Configure budgets, monitoring, quotas
  • Pilot with 3-5 users → full rollout
▶ SOP-3: Modify Model Access Cloud Platform
  • Evaluate cost impact (Opus ≈ 5x Sonnet per token), quota needs, region availability
  • Update SCP allowed model list if needed
  • Update managed-settings.json and Cowork MDM profiles
  • Adjust budget alerts for new cost profile
▶ SOP-4: Revoke User Access Identity Team
  • Remove user from Entra ID security group (SCIM auto-deprovisions)
  • For immediate revocation: delete user session in IAM Identity Center
  • Remove CLI + managed-settings and Cowork app from device
  • Verify via CloudTrail — no post-revocation API calls
▶ SOP-5: Decommission a Team Cloud Platform
  • Remove all users from Entra group
  • Delete permission set assignment
  • Disable Bedrock model access
  • Archive CloudTrail + invocation logs
  • Close/suspend AWS account (retain for audit period)
  • Remove MDM profiles + delete Entra group

Monitoring & Cost Management

Track usage, attribute costs, and enforce quotas.

▶ Cost Attribution (Account + Per-User)
  • Account-level: Per-team accounts → Cost Explorer shows each team’s spend
  • Per-user: Enable "IAM principal-based cost allocation" in Cost Explorer
  • Each Identity Center user gets unique role session → per-user spend visibility
  • ⚠️ Principal-based allocation is visibility only, not enforcement
▶ Quota Enforcement (OTel + Counter Store)
  • Claude Code & Cowork export OTel metrics natively (token usage, cost, sessions per user) Custom implementation needed
  • Data flow: Client → OTLP (port 4318) → OTel Collector (ECS Fargate) → CloudWatch/S3
  • Counter store (Redis): aggregate per-user token usage with TTL for auto-reset
  • Alert only: SNS notification when threshold exceeded (15 min lag)
  • Soft block: Lambda updates IAM deny policy (15-30 min)
  • Hard block: Proxy checks Redis pre-request (real-time)
▶ Recommended CloudWatch Alarms
  • InvocationThrottles > 10 in 5 minutes (quota exhaustion)
  • InvocationErrors > 5% error rate (configuration issues)
  • Monthly cost exceeds 80% of budget threshold

Security & Compliance

Data handling, guardrails, network security, and audit controls.

▶ Data Handling
  • All prompts/completions flow through AWS Bedrock — Anthropic does not store or access them
  • Data stays within AWS security boundary (your VPC/account)
  • Anthropic does NOT train models on your data via Bedrock
  • Zero Data Retention (ZDR) available — nothing stored after request completes
▶ Bedrock Guardrails
  • Create Guardrails in Bedrock console: content filtering, PII redaction, topic blocking
  • Safeguards: content filters, denied topics, word filters, sensitive information filters, contextual grounding
  • ⚠️ Do NOT enable automated reasoning policy — unsupported, causes runtime failures
  • Cross-account enforcement: AWS Organizations Bedrock policies apply guardrails across all team accounts
  • Create guardrail in management account → version it → attach resource-based policy → enable Bedrock Policy Type → attach to Claude OU
▶ Network Security
  • Consider AWS PrivateLink for Bedrock endpoints (keeps traffic off public internet)
  • Use sandbox network allowlists in Claude Code/Cowork
  • Ensure corporate firewall allows required AWS endpoint traffic
▶ Audit & Compliance
  • CloudTrail: complete audit trail with user identity
  • IAM Identity Center sessions traceable to Entra ID users
  • Bedrock invocation logs capture full request/response
  • Managed settings prevent bypassing organizational controls

Configure IAM Identity Center

Set up the native AWS identity path — no external IdP required.

Stakeholder: AWS Admin / Identity Team
  • Verify IAM Identity Center is enabled — Navigate to AWS IAM Identity Center console and confirm it's active in your management account
  • Provision users/groups — Ensure developers who need Claude access exist in IAM Identity Center (synced from AD/Okta/external source or manually created)
  • Create Permission Set for Bedrock access — Create a Permission Set with bedrock:InvokeModel* permissions scoped to target regions
  • Assign Permission Set to user/group — Assign the Permission Set to developer groups on the target AWS account
  • Set session duration — Recommended: 7 days (max: 90 days) to minimize re-authentication friction
  • Distribute SSO start URL — Share the SSO portal URL and profile name with developers (e.g., aws sso login --profile bedrock-dev)
💡 Key benefit: With IAM Identity Center, the CCWB solution auto-detects user identity from the AWSReservedSSO_* role ARN — full attribution (name, email, permission set) appears in the observability dashboard without any extra OIDC configuration.

Initialize Profile (ccwb init)

Clone the repo and run the interactive setup wizard to create a deployment profile.

Stakeholder: AWS Admin / Cloud Team
What is ccwb? — CCWB (Claude Code WorkBench) is the CLI tool from the AWS Solutions guidance repo. It orchestrates the full deployment lifecycle: init creates a local profile, deploy provisions CloudFormation stacks, package builds platform-specific installers, and distribute shares them with end users.

Clone & Install

git clone https://github.com/aws-solutions-library-samples/guidance-for-claude-code-with-amazon-bedrock
cd guidance-for-claude-code-with-amazon-bedrock/source
poetry install

Run the Wizard

poetry run ccwb init

Wizard Choices (IAM IDC Path)

  • Profile name → e.g. corp-bedrock-prod
  • Enable SSO authentication? → Select No (this refers to CCWB's own OIDC auth stack — select No because your users already authenticate via IAM Identity Center)
  • AWS Region → Region for CloudFormation stacks
  • Stack base name → Prefix for all stack names (e.g. claude-code-auth)
  • Enable monitoring? → Yes (recommended) — choose Central Collector or Sidecar
  • Enable analytics? → Yes for Athena + S3 data lake
  • Enable quota monitoring? → Yes to enforce per-user token limits
  • Enable Windows builds? → Yes if you have Windows users
  • Generate CoWork 3P MDM config? → Yes for Claude Desktop deployment
  • Claude model → Sonnet / Haiku / Opus
  • Cross-region profile → US / EU / APAC
  • Source region → e.g. us-east-1
📝 Note: ccwb init only saves configuration locally to ~/.ccwb/profiles/<name>.json. Nothing is deployed to AWS at this stage.

Deploy Infrastructure (ccwb deploy)

Provision AWS CloudFormation stacks based on your profile configuration.

Stakeholder: AWS Admin / Cloud Team
What ccwb deploy does under the hood: Reads your local profile and creates CloudFormation stacks in your AWS account. Since users already authenticate via IAM Identity Center (aws sso login), there's no need for CCWB to deploy its own auth stack (OIDC Provider / Cognito). It deploys only the monitoring, analytics, and quota stacks you enabled. 
# Deploy all configured stacks
poetry run ccwb deploy

# Monitor progress
poetry run ccwb status

What Gets Created (IAM IDC path)

  • Monitoring stack — VPC, ECS Fargate (OTEL collector), ALB, CloudWatch dashboards
  • Analytics stack — Kinesis Firehose → S3 → Glue → Athena (if enabled)
  • Quota stack — DynamoDB (limits), Lambda (monitor), SNS (alerts), API Gateway (if enabled)
  • CodeBuild stack — Windows .exe compilation (if enabled)
⏱️ Deployment time: 5–15 minutes depending on which stacks are enabled.

Build & Distribute Packages (ccwb package)

Build platform-specific installers and distribute to end users.

Stakeholder: AWS Admin / Cloud Team
What ccwb package does: Compiles the credential-helper binary for each target platform using Go cross-compilation. The binary handles token refresh, OTEL metric reporting, and AWS credential resolution. The resulting installers configure the user's ~/.aws/config with a credential_process entry pointing to this binary. 
# Build all platforms (recommended)
poetry run ccwb package --go --target-platform all

# Distribute via presigned S3 URLs
poetry run ccwb distribute

Platform Targets

  • macOS ARM64 (Apple Silicon M1/M2/M3/M4)
  • macOS Intel (x86_64) — runs on all Macs via Rosetta
  • Linux x86_64 and ARM64
  • Windows x64 (via CodeBuild + Nuitka)

Distribution Options

  • Presigned S3 URLs — Time-limited download links shared via Slack/email
  • Authenticated Landing Page — Self-service portal with SSO login + platform detection

Quota Management

Enforce per-user and per-group token limits with alerts and blocking.

Stakeholder: Platform / FinOps Team

How It Works

  • DynamoDB table — Stores per-user/group/default token limits
  • Lambda (every 15 min) — Checks thresholds via PromQL, sends SNS alerts at 80%, 90%, 100%
  • API Gateway — Real-time quota check at credential issuance time

Configuration Options

  • Monthly token limit — Default: 225M tokens/user/month
  • Burst buffer — 5–25% above daily average (default 10%)
  • Daily enforcement — alert (notify only) or block (deny credentials)
  • Monthly enforcement — alert or block (recommended: block)
  • Check interval — 0 min (every request, +200ms) / 30 min (default) / 60 min
⚠️ Note: Per-user quota enforcement requires the monitoring stack to be enabled. Without monitoring, only aggregate limits are possible.
